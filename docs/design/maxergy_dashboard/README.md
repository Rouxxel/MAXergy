# MAXergy benchmark and personalised dashboard

## 1. Overview

The MAXergy dashboard is generated by a deterministic pipeline:

```
Input JSON
→ Production energy model (scripts/energy_model/pipeline.py)
→ Output JSON (all six upgrade scenarios × three price scenarios)
→ Recommendation logic (select_best_scenario)
→ Presentation data (headline, monthly series, cumulative savings, Year 1 summary)
→ Dashboard components
→ Optional LLM explanation
```

The model is the **single source of truth** for all numbers. The frontend and any LLM integration must read values from the model output JSON. Neither the frontend nor the LLM may independently recalculate:

- Energy consumption or generation
- Energy costs (grid, heating, mobility)
- Upgrade investment costs
- Financing instalments, repayment totals, or interest
- Net savings or energy-cost reductions
- Cumulative net savings or break-even timing
- The recommended upgrade plan

The model is invoked at request time using either the CLI (`run_model.py`) or the Python API (`compute_model()`). Output values are then read directly from the returned JSON without any further arithmetic in the presentation layer, except for formatting, aggregation of pre-computed monthly values into display series, and computing monthly averages from annual totals already present in the output (e.g. `annual_total / 12`).

`compute_model()` returns the complete JSON-serialisable result dictionary without any UI formatting. The returned dict has the same structure as the output file written by `run_model()`. The function raises `ValueError` for invalid input.

## 5. Data processing flow

### Step 1: Input validation

`validate_and_parse()` in `scripts/energy_model/input_validator.py` validates and normalises the raw input dict into a typed `ParsedInput` dataclass. It applies German defaults where values are missing, records provenance for each field, and raises `ValueError` for invalid input.

Fields validated and normalised:

- **Household**: postcode, country, occupant count
- **Electricity**: annual kWh, Arbeitspreis, Grundpreis, contract end date
- **Heating**: fuel type (`gas` or `oil`), annual consumption, price per unit
- **Mobility**: vehicle type, annual mileage, consumption per 100 km, effective petrol price
- **Roof and solar**: usable area, orientation, tilt, shading factor → kWp derivation
- **Financing**: loan term (years), annual rate
- **Forecast horizons**: short-term months, long-term years

### Step 2: Baseline calculation

The model distributes annual consumption across months using seasonal profiles from `scripts/energy_model/consumption.py` (BDEW-style load fractions). For each month it computes electricity, heating, and mobility costs under the active price model. The baseline represents the household's total energy cost without any upgrade.

### Step 3: Upgrade scenarios

Six scenarios are evaluated by `ScenarioOrchestrator` in `scripts/energy_model/orchestrator.py`:

| Scenario key | Description |
|--------------|-------------|
| `solar_only` | PV system, no battery, no heating or mobility upgrade |
| `pv_battery` | PV system + home battery storage |
| `pv_heatpump` | PV system + heat pump (replaces gas boiler) |
| `pv_ev` | PV system + home EV charger (replaces petrol vehicle) |
| `pv_battery_heatpump` | PV + battery + heat pump |
| `full_upgrade` | PV + battery + heat pump + EV charger |

Each scenario is computed at `UpgradeEnergyModel` level in `scripts/energy_model/upgrade_model.py`, which dispatches PV generation, battery charge/discharge, heat pump demand, and EV home-charging demand using the `SCENARIO_FLAGS` dict.

### Step 4: Financing

`FinancingModel` in `scripts/energy_model/financing.py` computes a standard fixed-rate annuity loan for each scenario independently.

Per scenario, the model produces:

- **Gross investment** — sum of technology component costs (PV, battery, heat pump, EV charger)
- **Subsidy** — deducted from gross investment (default: €0; explicit user input required to apply)
- **Upfront contribution** — user cash contribution (default: €0)
- **Financed principal** — gross investment minus subsidy minus upfront contribution
- **Monthly instalment** — standard annuity formula at the configured rate
- **Total repayment** — instalment × loan term months
- **Financing end date** — derived from loan term; payments drop to zero beyond this month

### Step 5: Net savings

The model computes net savings as follows:

```
upgraded total household cost (monthly)
  = upgraded energy cost
  + monthly financing instalment
```

```
net monthly savings
  = baseline energy cost
  − upgraded total household cost
```

```
cumulative net savings
  = running total of net monthly (or annual) savings from start of forecast
```

For the long-term projection (annual aggregates):

```
annual net savings
  = annual energy cost reduction
  − annual financing payments
```

Energy cost reduction is the difference between what the household would have paid under the baseline and what they actually pay with the upgrade active. Financing payments drop to zero after the loan term ends.

### Step 6: Recommendation

The recommended scenario is selected deterministically in `select_best_scenario()` in `scripts/create_app_visualizations.py`:

```python
max(
    output["upgrade_scenarios"],
    key=lambda sn: (
        output["upgrade_scenarios"][sn]["long_term_projection"]["central"][-1]
            ["financial_result"]["cumulative_net_savings_eur"],          # primary
        output["upgrade_scenarios"][sn]["long_term_projection"]["central"][0]
            ["financial_result"]["annual_net_savings_eur"],              # tie-breaker 1
        -output["upgrade_scenarios"][sn]["financing"]["monthly_instalment_eur"],  # tie-breaker 2
    ),
)
```

**Criteria in order**:

1. **Highest central 20-year cumulative net savings** (primary criterion)
2. **Higher Year 1 annual net savings** (first tie-breaker)
3. **Lower monthly financing instalment** (second tie-breaker)

The LLM must not select the plan independently. It receives the deterministic recommendation result and may only explain it.

For the benchmark household, the recommended scenario is **`full_upgrade`** with a central 20-year cumulative net saving of **€24,828** and a break-even in Year 8.

## 6. Short-term estimate and long-term projection

The dashboard presents two distinct result types that use different price models.

### Short-term estimate (months 1–12)

- Covers the configured `short_term_months` (default: 12 months)
- Uses `ConstantShortTermPriceModel`: prices are held constant at the user's current tariff
- Validated against Destatis rolling-origin backtest as the most accurate short-horizon approach
- Shows exact monthly cost figures where the user's contracted electricity price applies
- Provides month-by-month estimates of baseline cost, upgraded cost, financing payment, and net savings

### Long-term projection (years 1–20)

- Covers the configured `long_term_years` (default: 20 years)
- Uses `ScenarioPriceModel`: deterministic annual price trends per scenario
- Three scenarios: **low** (conservative trends), **central** (moderate trends), **high** (accelerated trends)
- Annual trend assumptions (from model output `price_models.long_term.assumptions`):
  - Electricity: low 1%, central 3%, high 5% p.a.
  - Gas: low 1%, central 4%, high 7% p.a.
  - Petrol: low 1%, central 3%, high 5% p.a.
- **These are not statistical confidence intervals**. They represent policy-range assumptions for user communication.
- Must be described as a **projection**, not a guaranteed forecast.

The central projection drives the main headline number and the recommendation. The low and high scenarios communicate sensitivity to energy price assumptions.

The short-term and long-term sections intentionally use different price models. A connector element in the monthly chart signals this discontinuity visually.

## 11. Personalised page behaviour

After the visitor completes the questionnaire:

1. Replace the benchmark input JSON with the user-supplied input JSON.
2. Run the same production model: `compute_model(user_raw_input)`.
3. Generate a user-specific output JSON with identical schema.
4. Run `select_best_scenario(user_output)` to determine the recommended scenario.
5. Build the same presentation data structure using user-specific model output.
6. Render all dashboard components using user data, with no residual benchmark values.

**Page mode:** `personalised`

**Suggested heading:**

> Your estimated energy upgrade results

**Suggested subtitle:**

> Based on the information you provided about your home and energy use.

**Behaviour changes from benchmark mode:**

| Element | Benchmark | Personalised |
|---------|-----------|-------------|
| Benchmark badge | Visible | Hidden |
| Personalisation CTA | Prominent | Hidden or secondary |
| Household summary cards | Show benchmark profile values | Show user-supplied values |
| Hero savings number | Benchmark result | User-specific result |
| Chart data | Benchmark model output | User-specific model output |
| Assumptions disclosure | Show benchmark defaults | Show which user-supplied values were used and which were defaulted |
| Warnings | Show benchmark warnings | Show user-specific model warnings |

The interface structure (component layout, chart types, Year 1 summary, recommendation card) does not change between modes. Only the data and explanatory text change.

## 12. LLM interaction

The LLM's role is **explanatory only**. It receives the deterministic model output and recommendation result, then translates numbers into plain-language explanations.

**The LLM may:**

- Explain why the recommended scenario has the highest projected 20-year savings
- Summarise the main result in plain language
- Describe the mechanics of the break-even timeline
- Explain why Year 1 net savings are negative (financing costs exceed early energy savings)
- Describe what happens when financing ends (monthly outflow decreases substantially)
- Explain the low / central / high projection band and why the range exists
- Mention specific assumptions and warnings from the model output
- Answer user questions using numbers from the model output

**The LLM must not:**

- Recompute energy savings, costs, or investment totals
- Modify or override any numerical result from the model
- Invent subsidy eligibility not present in the model output
- Invent roof suitability or solar potential not present in the model output
- Recalculate financing or present different instalment figures
- Choose a different recommended plan than the one determined by `select_best_scenario()`
- Present long-term projections as guaranteed savings or fixed outcomes

> The Full Upgrade — solar panels, a battery, a heat pump, and an EV charger — has the highest projected 20-year cumulative net savings for this benchmark household under the central energy price scenario.
>

This reverses over time as energy prices rise while the loan repayment stays fixed. Under central assumptions, the household reaches break-even around Year 8.
>

> These figures describe the benchmark profile. Two defaults were applied: solar system size was estimated from roof area (5.83 kWp), and battery capacity used the standard assumption (10 kWh). Your personalised estimate may differ.

## 15. Known limitations

The following limitations are inherent to the current model and must be communicated to users, particularly in the assumptions disclosure:

| Area | Limitation |
|------|-----------|
| Solar potential | Regional postcode-level irradiance, not property-level shading or obstruction analysis |
| Roof characteristics | User-supplied or estimated from area; no satellite verification |
| Battery dispatch | Monthly balance model, not sub-hourly or daily dispatch simulation |
| Energy prices (long-term) | Scenario trend assumptions, not econometric forecasts; wide uncertainty at 20 years |
| Financing | User-supplied rate and term; actual loan offers may differ. No credit assessment. |
| Subsidies | No subsidy is assumed by default. Eligibility depends on programme rules, household circumstances, and application outcome — not guaranteed. |
| Technology costs | Default unit costs (€1,400/kWp, €600/kWh battery, €12,000 heat pump, €1,200 charger) are representative; actual quotes may differ. |
| EV purchase cost | Not included by default. `ev_purchase_eur = 0` means the model costs only the home charger. Financing of a vehicle purchase requires explicit user input. |
| Technology lifetime | No replacement or maintenance costs are modelled within the 20-year horizon. |
| PV degradation | A constant 0.5% annual yield degradation is applied; real-world degradation varies by panel technology. |
| Results are estimates | All outputs are estimates for planning purposes, not quotations or financial advice. |
